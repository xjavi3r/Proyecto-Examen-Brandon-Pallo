CRUD de Inventario y Ventas

Cuatro tipos de mantenimiento aplicados al CRUD

Correctivo, adaptativo, perfectivo y preventivo sobre la entidad Producto, manteniendo la integracion con el Event Manager mediante POST /events.

Proyecto Django: app productos

El mantenimiento se hizo solo sobre el CRUD de productos.

Resumen

Cada mantenimiento resolvio una necesidad distinta del mismo CRUD.

Tipo Accion realizada Por que aplica
Correctivo Corregir el POST API para guardar descripcion. Habia un campo existente que no se persistia al crear por API.
Adaptativo Agregar ubicacion al producto. El inventario necesita adaptarse a bodega, estante o mostrador.
Perfectivo Mostrar descripcion y ubicacion en la interfaz. Mejora la lectura y uso del CRUD sin cambiar su objetivo.
Preventivo Normalizar nombre/ubicacion y bloquear nombre vacio. Evita errores futuros al crear o actualizar productos.
Todos los cambios pertenecen al CRUD de Producto
Mantenimiento Correctivo

Se corrigio la creacion por API para no perder datos.

Problema

El modelo ya tenia el campo descripcion, pero al crear por POST /api/productos/ no se asignaba.

Resultado: el CRUD aceptaba productos, pero la descripcion se perdia si se usaba la API.

Correccion

Se agrego descripcion al constructor de Producto en productos_list_create.

Tambien se agrego ubicacion para que el nuevo campo adaptativo funcione desde API.

Archivo: productos/views.py

Mantenimiento Adaptativo

El CRUD se adapto a una necesidad real de inventario: ubicacion fisica.

Necesidad nueva

En un inventario real no basta con saber que un producto existe.

Tambien importa saber donde esta: bodega, estante, mostrador o zona de venta.

Cambio aplicado

Se agrego el campo ubicacion al modelo, formulario, API y diccionario JSON.

Esto adapta el CRUD sin cambiar su arquitectura.

Archivos: models.py, forms.py, views.py, migracion 0003

Mantenimiento Preventivo

Se previenen errores de datos antes de que lleguen a la base.

Riesgo

Un producto podria llegar con nombre vacio o con espacios innecesarios desde la API.

Eso causa busquedas inconsistentes y registros dificiles de mantener.

Prevencion

Se agrego clean() en el modelo para normalizar nombre y ubicacion.

Si el nombre queda vacio, se bloquea el guardado con una validacion.

Archivo: productos/models.py
Preventivo - Codigo

Antes y despues de las validaciones del modelo.

Antes
def __str__(self):
    return f"{self.nombre} ({self.categoria})"

def to_dict(self):
    return {
        "id": self.id,
        "nombre": self.nombre,
        "descripcion": self.descripcion,
        "categoria": self.categoria,
    }
Despues
def __str__(self):
    return f"{self.nombre} ({self.categoria})"

def clean(self):
    super().clean()
    self.nombre = (self.nombre or "").strip()
    self.ubicacion = (self.ubicacion or "").strip()

    if not self.nombre:
        raise ValidationError({
            "nombre": "El nombre no puede estar vacio."
        })
Beneficio: datos mas limpios y menos errores futuros
Integracion

El CRUD mantiene el envio de eventos al Event Manager.

enviar_evento_async(
    "CREATE",

Por que importa

Cada accion del CRUD sigue enviando un evento por POST /events.

Los cambios de mantenimiento no rompen la integracion con el EPN Event Manager.

Archivo: productos/event_client.py
Cierre

Conclusion para la exposicion

Se aplicaron cuatro mantenimientos especificamente al CRUD de Producto. El correctivo soluciono una perdida de datos en la API; el adaptativo agrego ubicacion para una necesidad real de inventario; el perfectivo mejoro la visualizacion y captura de informacion; y el preventivo agrego validaciones para evitar datos inconsistentes. El CRUD conserva sus operaciones basicas y la integracion con el Event Manager.
